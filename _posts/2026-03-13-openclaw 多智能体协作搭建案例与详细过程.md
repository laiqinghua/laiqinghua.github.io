## 引言

在 AI 助理系统日益普及的今天，单一智能体往往难以应对复杂的工作场景。OpenClaw 作为一个自托管网关系统，提供了强大的多智能体协作能力，让你能够在同一台服务器上运行多个隔离的 AI 助理，并根据不同场景自动路由消息。本文将通过实际案例，详细讲解如何搭建和配置 OpenClaw 多智能体协作系统。

## 什么是 OpenClaw

OpenClaw 是一个自托管网关，连接 WhatsApp、Telegram、Discord、iMessage 等聊天应用到 AI 编码助手。它运行在你的硬件上，支持多通道、多智能体路由、媒体处理和完整的会话管理。

核心特点包括：

- 自托管：运行在你的硬件上，数据完全可控
- 多通道：单个网关同时服务 WhatsApp、Telegram、Discord 等多个平台
- 多智能体路由：根据发送者、群组、频道等条件将消息路由到不同的 AI 助理
- 开源：MIT 许可，社区驱动

## 多智能体架构核心概念

### 什么是"一个智能体"

在 OpenClaw 中，一个智能体是一个完全独立的大脑，拥有：

- 独立的工作空间：包含 AGENTS.md、SOUL.md、USER.md 等配置文件
- 独立的状态目录：存储认证信息、模型注册表和配置
- 独立的会话存储：聊天记录和路由状态存储在 ~/.openclaw/agents/<agentId>/sessions

### 关键路径映射

- 配置文件：~/.openclaw/openclaw.json
- 状态目录：~/.openclaw
- 工作空间：~/.openclaw/workspace
- 智能体目录：~/.openclaw/agents/<agentId>/agent
- 会话存储：~/.openclaw/agents/<agentId>/sessions

### 路由绑定机制

绑定（Binding）是入站消息路由到智能体的核心机制，遵循"最具体优先"原则：

1. peer 匹配（精确的 DM/群组/频道 ID）
2. parentPeer 匹配（线程继承）
3. guildId + roles（Discord 角色路由）
4. guildId（Discord 服务器）
5. teamId（Slack）
6. accountId 匹配
7. 通道级匹配
8. 回退到默认智能体

## 实战案例一：WhatsApp 日常聊天 + Telegram 深度工作

这个案例展示如何根据不同通道路由到不同智能体，WhatsApp 用于快速日常交流，Telegram 用于深度工作。

### 配置步骤

第一步：创建两个智能体工作空间

```bash
openclaw agents add chat
openclaw agents add opus
```

每个智能体获得独立的工作空间，包含 SOUL.md、AGENTS.md 和可选的 USER.md，以及专用的 agentDir 和会话存储。

第二步：创建通道账户

为每个智能体在首选通道上创建账户：

- Discord：每个智能体一个机器人，启用 Message Content Intent，复制每个令牌
- Telegram：通过 BotFather 为每个智能体创建机器人，复制每个令牌
- WhatsApp：为每个账户链接不同的手机号码

```bash
openclaw channels login --channel telegram --account chat
openclaw channels login --channel telegram --account opus
```

第三步：配置路由绑定

编辑 ~/.openclaw/openclaw.json：

```json
{
  "agents": {
    "list": [
      {
        "id": "chat",
        "name": "Everyday",
        "workspace": "~/.openclaw/workspace-chat",
        "model": "anthropic/claude-sonnet-4-5"
      },
      {
        "id": "opus",
        "name": "Deep Work",
        "workspace": "~/.openclaw/workspace-opus",
        "model": "anthropic/claude-opus-4-6"
      }
    ]
  },
  "bindings": [
    { "agentId": "chat", "match": { "channel": "whatsapp" } },
    { "agentId": "opus", "match": { "channel": "telegram" } }
  ]
}
```

第四步：重启并验证

```bash
openclaw gateway restart
openclaw agents list --bindings
openclaw channels status --probe
```

## 实战案例二：同一通道内按联系人路由

保持 WhatsApp 在快速智能体上，但将特定联系人路由到 Opus 深度工作智能体。

```json
{
  "agents": {
    "list": [
      {
        "id": "chat",
        "name": "Everyday",
        "workspace": "~/.openclaw/workspace-chat",
        "model": "anthropic/claude-sonnet-4-5"
      },
      {
        "id": "opus",
        "name": "Deep Work",
        "workspace": "~/.openclaw/workspace-opus",
        "model": "anthropic/claude-opus-4-6"
      }
    ]
  },
  "bindings": [
    {
      "agentId": "opus",
      "match": { 
        "channel": "whatsapp", 
        "peer": { "kind": "direct", "id": "+8613800138000" } 
      }
    },
    { "agentId": "chat", "match": { "channel": "whatsapp" } }
  ]
}
```

注意：peer 绑定始终优先于通道级规则，因此将其放在配置的前面。

## 实战案例三：家庭智能体绑定到 WhatsApp 群组

为家庭群组配置专用的智能体，设置提及门控和更严格的工具策略。

```json
{
  "agents": {
    "list": [
      {
        "id": "family",
        "name": "Family",
        "workspace": "~/.openclaw/workspace-family",
        "identity": { "name": "Family Bot" },
        "groupChat": {
          "mentionPatterns": ["@family", "@familybot", "@Family Bot"]
        },
        "sandbox": {
          "mode": "all",
          "scope": "agent"
        },
        "tools": {
          "allow": [
            "exec",
            "read",
            "sessions_list",
            "sessions_history",
            "sessions_send",
            "sessions_spawn",
            "session_status"
          ],
          "deny": ["write", "edit", "apply_patch", "browser", "canvas", "nodes", "cron"]
        }
      }
    ]
  },
  "bindings": [
    {
      "agentId": "family",
      "match": {
        "channel": "whatsapp",
        "peer": { "kind": "group", "id": "120363999999999999@g.us" }
      }
    }
  ]
}
```

注意事项：

- 工具允许/拒绝列表针对的是工具，不是技能
- 对于更严格的门控，设置 groupChat.mentionPatterns 并启用群组允许列表

## 子智能体系统详解

### 什么是子智能体

子智能体是从现有智能体运行中派生的后台智能体运行。它们在独立的会话中运行，完成后将结果通告回请求者聊天频道。

### 使用场景

- 并行化研究、长任务或慢速工具工作，不阻塞主运行
- 保持子智能体默认隔离（会话分离 + 可选沙盒）
- 支持可配置的嵌套深度，用于编排器模式

### 基本使用

使用 sessions_spawn 工具启动子智能体：

```javascript
sessions_spawn({
  task: "研究最新的前端框架趋势",
  label: "research-task",
  model: "cheaper-model",
  runTimeoutSeconds: 600
})
```

### 线程绑定会话

当通道启用线程绑定时，子智能体可以保持绑定到线程，这样该线程中的后续用户消息将继续路由到同一个子智能体会话。

目前支持的通道：

- Discord：支持持久线程绑定子智能体会话、手动线程控制

快速流程：

1. 使用 sessions_spawn 并设置 thread: true
2. OpenClaw 创建或绑定线程到该会话目标
3. 该线程中的回复和后续消息路由到绑定的会话
4. 使用 /session idle 检查/更新不活动自动取消聚焦
5. 使用 /unfocus 手动分离

### 嵌套子智能体

默认情况下，子智能体不能派生自己的子智能体（maxSpawnDepth: 1）。可以通过设置 maxSpawnDepth: 2 启用一级嵌套，实现编排器模式：主智能体 → 编排器子智能体 → 工作者子子智能体。

启用配置：

```json
{
  "agents": {
    "defaults": {
      "subagents": {
        "maxSpawnDepth": 2,
        "maxChildrenPerAgent": 5,
        "maxConcurrent": 8,
        "runTimeoutSeconds": 900
      }
    }
  }
}
```

深度级别说明：

- 深度 0：主智能体，始终可以派生
- 深度 1：子智能体（当 maxSpawnDepth >= 2 时为编排器），仅在 maxSpawnDepth >= 2 时可以派生
- 深度 2：子子智能体（叶子工作者），从不可以派生

## 每智能体沙盒和工具配置

从 v2026.1.6 开始，每个智能体可以拥有自己的沙盒和工具限制。

```json
{
  "agents": {
    "list": [
      {
        "id": "personal",
        "workspace": "~/.openclaw/workspace-personal",
        "sandbox": {
          "mode": "off"
        }
      },
      {
        "id": "family",
        "workspace": "~/.openclaw/workspace-family",
        "sandbox": {
          "mode": "all",
          "scope": "agent",
          "docker": {
            "setupCommand": "apt-get update && apt-get install -y git curl"
          }
        },
        "tools": {
          "allow": ["read"],
          "deny": ["exec", "write", "edit", "apply_patch"]
        }
      }
    ]
  }
}
```

优势：

- 安全隔离：限制不受信任智能体的工具
- 资源控制：沙盒特定智能体，同时保持其他智能体在主机上运行
- 灵活策略：每个智能体不同的权限

## 工具组快捷方式

工具策略支持 group:* 条目，可扩展为多个工具。在 tools.allow/tools.deny 中使用：

- group:runtime：exec、bash、process
- group:fs：read、write、edit、apply_patch
- group:sessions：sessions_list、sessions_history、sessions_send、sessions_spawn、session_status
- group:memory：memory_search、memory_get
- group:web：web_search、web_fetch
- group:ui：browser、canvas
- group:automation：cron、gateway
- group:messaging：message
- group:nodes：nodes
- group:openclaw：所有内置 OpenClaw 工具

示例（仅允许文件工具 + 浏览器）：

```json
{
  "tools": {
    "allow": ["group:fs", "browser"]
  }
}
```

## 认证管理

子智能体认证按智能体 ID 解析，不是按会话类型：

- 子智能体会话键为 agent:<agentId>:subagent:<uuid>
- 认证存储从该智能体的 agentDir 加载
- 主智能体的认证配置文件作为回退合并；冲突时智能体配置文件优先

注意：合并是附加的，主配置文件始终可用作回退。目前不支持完全隔离的每智能体认证。

## 通告机制

子智能体通过通告步骤报告结果：

1. 通告步骤在子智能体会话内部运行
2. 如果子智能体回复 ANNOUNCE_SKIP，不发布任何内容
3. 交付取决于请求者深度：
   - 顶级请求者会话使用 follow-up agent 调用进行外部交付
   - 嵌套请求者子智能体会话接收内部 follow-up 注入
4. 通告回复在可用时保留线程/主题路由

通告负载包括：

- 运行时间
- 令牌使用量（输入/输出/总计）
- 配置模型定价时的估算成本
- sessionKey、sessionId 和转录路径

## 常见问题

### 多个智能体共享同一个 WhatsApp 号码

可以在一个 WhatsApp 账户上路由不同的 WhatsApp 私聊到不同的智能体。通过发送者 E.164 号码匹配，使用 peer.kind: "direct"。回复仍然来自同一个 WhatsApp 号码。

重要细节：私聊会折叠到智能体的主会话键，因此真正的隔离需要每人一个智能体。

### 如何限制子智能体的工具使用

通过 tools.subagents.tools 配置：

```json
{
  "tools": {
    "subagents": {
      "tools": {
        "deny": ["gateway", "cron"]
      }
    }
  }
}
```

### 子智能体并发控制

子智能体使用专用的进程内队列通道：

- 通道名称：subagent
- 并发数：agents.defaults.subagents.maxConcurrent（默认 8）

### 如何停止子智能体

- 在请求者聊天中发送 /stop 会中止请求者会话并停止从中派生的任何活动子智能体运行，级联到嵌套子智能体
- /subagents kill <id> 停止特定子智能体并级联到其子智能体

## 总结

OpenClaw 的多智能体协作系统提供了灵活而强大的架构，让你能够：

- 根据不同通道、联系人、群组路由消息到专用智能体
- 为不同场景配置不同的模型和工具策略
- 使用子智能体并行处理复杂任务
- 通过沙盒和工具限制确保安全

通过合理配置，你可以构建一个既高效又安全的 AI 助理团队，每个成员专注于自己最擅长的工作领域。
