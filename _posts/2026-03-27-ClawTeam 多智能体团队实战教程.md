## 01 为什么需要 ClawTeam

现在的 AI Agent 很强大，但它们都在**单打独斗**。

当你面对复杂任务时，不得不手动协调多个 Agent，来回切换上下文，最后把碎片化的结果拼在一起。整个过程繁琐又容易出错。

**如果 Agent 能像人类团队一样协作呢？**

ClawTeam 就是干这个的。它让 AI Agent 自动组成团队，智能分工，实时共享信息，动态调整策略，最终交出完整成果。

你只需要告诉 Leader Agent 目标，剩下的交给智能体 swarm 自己 orchestrate。

## 02 核心概念

在开始之前，先理解几个关键概念：

**Leader Agent（领导智能体）**
- 团队的协调者，负责 spawn 子 Agent、分配任务、监控进度
- 通常由你来扮演，或者让一个高级 Agent 担任

**Worker Agent（工作智能体）**
- 执行具体任务的子 Agent
- 每个 Worker 都有独立的工作环境（git worktree + tmux 窗口）

**Team（团队）**
- 一组协作的 Agent 集合
- 有共享的任务看板、收件箱、工作区

**Task（任务）**
- 有状态的工作项：pending → in_progress → completed/blocked
- 支持依赖链：任务 B 可以依赖任务 A，A 完成后 B 自动解锁

**Inbox（收件箱）**
- Agent 之间的点对点消息系统
- 支持发送、接收、广播

## 03 安装与准备

### 安装 ClawTeam

```bash
pip install clawteam

# 或者从源码安装
git clone https://github.com/HKUDS/ClawTeam.git
cd ClawTeam
pip install -e .

# 可选：P2P 传输支持（ZeroMQ）
pip install -e ".[p2p]"
```

### 环境要求

- Python 3.10+
- tmux（用于交互式监控）
- 任意 CLI Agent（Claude Code、Codex、OpenClaw 等）

### 安装前检查

```bash
# 检查 tmux
tmux -V

# 检查 ClawTeam
clawteam --help

# 检查你的 Agent CLI（以 claude 为例）
claude --version
```

**重要**：如果 Agent CLI 单独运行都有问题，ClawTeam 也帮不了你。先确保基础环境正常。

## 04 快速开始：手动驱动团队

### 第一步：创建团队

```bash
# 创建一个名为 my-team 的团队，你担任 leader
clawteam team spawn-team my-team -d "构建认证模块" -n leader
```

这个命令会：
- 在 `~/.clawteam/teams/` 创建团队配置
- 初始化共享的任务看板和收件箱
- 创建主 git 工作区

### 第二步：Spawn 工作 Agent

```bash
# Spawn 第一个 Worker：负责实现 OAuth2 流程
clawteam spawn --team my-team --agent-name alice --task "实现 OAuth2 认证流程"

# Spawn 第二个 Worker：负责写单元测试
clawteam spawn --team my-team --agent-name bob --task "为认证模块编写单元测试"
```

每个 Worker 会自动获得：
- 独立的 git worktree（分支名：`clawteam/my-team/alice`）
- 独立的 tmux 窗口
- 自动注入的协调协议提示词

### 第三步：Worker 自动学会协作

被 spawn 的 Agent 会自动收到一段协调协议提示词，教会它们：

```bash
# 查看分配给自己的任务
clawteam task list my-team --owner alice

# 开始任务时更新状态
clawteam task update my-team <任务 ID> --status in_progress

# 完成任务时
clawteam task update my-team <任务 ID> --status completed

# 给 Leader 发消息
clawteam inbox send my-team leader "OAuth2 实现完成，测试通过"

# 给队友发消息
clawteam inbox send my-team bob "API 文档在这里：..."

# 检查收件箱
clawteam inbox receive my-team

# 报告空闲（任务完成后）
clawteam lifecycle idle my-team
```

**关键点**：这些命令是自动注入到 Agent 的提示词中的，不需要你手动教。任何能执行 shell 命令的 CLI Agent 都能参与协作。

### 第四步：监控团队工作

```bash
# 方式 1：终端看板
clawteam board show my-team

# 方式 2：自动刷新仪表盘（每 3 秒）
clawteam board live my-team --interval 3

# 方式 3：tmux 平铺视图（同时看到所有 Agent 的工作界面）
clawteam board attach my-team

# 方式 4：Web UI（浏览器访问）
clawteam board serve --port 8080
```

**推荐**：用 `board attach` 可以看到所有 Agent 在 tmux 中的实时工作画面，非常直观。

## 05 任务管理与依赖

### 创建带依赖的任务

```bash
# 创建任务，指定负责人
clawteam task create my-team "设计 REST API 架构" -o architect

# 创建依赖任务：任务 2 依赖任务 1
clawteam task create my-team "实现 JWT 认证" -o backend1 --blocked-by 1

# 创建多依赖任务：任务 3 依赖任务 1 和 2
clawteam task create my-team "集成测试" -o tester --blocked-by 1,2
```

### 任务状态流转

```
pending → in_progress → completed
                     ↘ blocked
```

**自动解锁**：当任务 1 完成后，所有依赖任务 1 的任务会自动从 `blocked` 变为 `pending`。

### 查看任务

```bash
# 查看所有任务
clawteam task list my-team

# 按状态过滤
clawteam task list my-team --status blocked

# 按负责人过滤
clawteam task list my-team --owner backend1

# JSON 输出（方便脚本处理）
clawteam task list my-team --json
```

### 等待任务完成

```bash
# 阻塞等待所有任务完成（超时 300 秒）
clawteam task wait my-team --timeout 300
```

## 06 Agent 间通信

### 点对点消息

```bash
# 发送消息给特定 Agent
clawteam inbox send my-team architect "API 规范已更新，请查看"

# 接收消息（会消耗掉）
clawteam inbox receive my-team

# 查看消息但不消耗
clawteam inbox peek my-team
```

### 广播消息

```bash
# 给所有团队成员发消息
clawteam inbox broadcast my-team "所有人注意：下午 3 点同步进度"
```

### 实际协作场景

```bash
# 架构师完成后通知后端
clawteam inbox send my-team backend1 "OpenAPI 规范：/api/auth/* 端点已定义"

# 后端完成后通知测试
clawteam inbox send my-team tester "认证端点已就绪，可以开始测试"

# 测试完成后汇报给 Leader
clawteam inbox send my-team leader "全部 47 个测试通过 ✅"
```

## 07 使用模板一键启动团队

ClawTeam 支持用 TOML 文件定义团队模板，一条命令启动完整团队。

### 内置模板：AI 对冲基金

```bash
# 一键启动 7 人投资分析团队
clawteam launch hedge-fund --team fund1 --goal "分析 AAPL、MSFT、NVDA 2026 年 Q2 表现"
```

自动 spawn 的 Agent：
- 📊 投资组合经理（Leader）
- 🎩 巴菲特分析师（价值投资）
- 🚀 增长分析师（颠覆性技术）
- 📈 技术分析师（EMA、RSI、布林带）
- 📋 基本面分析师（财务比率）
- 📰 情绪分析师（新闻 + 内幕交易）
- 🛡️ 风险管理师

### 查看可用模板

```bash
clawteam template list
```

### 创建自定义模板

在 `~/.clawteam/templates/` 创建 TOML 文件：

```toml
[team]
name = "web-dev-team"
description = "全栈 Web 开发团队"

[roles.architect]
prompt = "你是一位资深软件架构师，负责设计系统架构和 API 规范"

[roles.backend]
prompt = "你是一位后端开发工程师，负责实现业务逻辑和数据库"

[roles.frontend]
prompt = "你是一位前端开发工程师，负责实现用户界面"

[roles.tester]
prompt = "你是一位测试工程师，负责编写和执行测试用例"

[tasks]
- role = "architect"
  subject = "设计 REST API 架构"
- role = "backend"
  subject = "实现用户认证模块"
  blocked_by = [1]
- role = "frontend"
  subject = "实现登录页面"
  blocked_by = [1]
- role = "tester"
  subject = "编写集成测试"
  blocked_by = [2,3]
```

## 08 实战案例：全栈应用开发

### 场景描述

目标：构建一个带认证、数据库、React 前端的全栈待办应用。

### Leader Agent 的完整工作流

```bash
# 1. 创建团队
clawteam team spawn-team webapp -d "全栈待办应用" -n leader

# 2. 创建任务链
clawteam task create webapp "设计 REST API 架构" -o architect
clawteam task create webapp "实现 JWT 认证" -o backend1 --blocked-by 1
clawteam task create webapp "构建 PostgreSQL 数据层" -o backend2 --blocked-by 1
clawteam task create webapp "实现 React 前端" -o frontend
clawteam task create webapp "集成测试" -o tester --blocked-by 2,3,4

# 3. Spawn 子 Agent
clawteam spawn tmux claude --team webapp --agent-name architect --task "设计 REST API 架构，输出 OpenAPI 规范"
clawteam spawn tmux claude --team webapp --agent-name backend1 --task "实现 JWT 认证模块"
clawteam spawn tmux claude --team webapp --agent-name backend2 --task "构建 PostgreSQL 数据模型"
clawteam spawn tmux claude --team webapp --agent-name frontend --task "用 React 实现前端 UI"
clawteam spawn tmux claude --team webapp --agent-name tester --task "编写 pytest 集成测试"

# 4. 监控进度
clawteam board live webapp --interval 3

# 5. 等待所有任务完成
clawteam task wait webapp --timeout 600

# 6. 合并所有工作区到主分支
clawteam workspace merge webapp architect
clawteam workspace merge webapp backend1
clawteam workspace merge webapp backend2
clawteam workspace merge webapp frontend

# 7. 清理工作区
clawteam workspace cleanup webapp architect
clawteam workspace cleanup webapp backend1
# ... 清理其他
```

### 关键优势

**并行开发**：5 个 Agent 同时工作，而不是串行

**自动依赖管理**：后端任务自动等待架构完成，测试自动等待所有开发完成

**零冲突**：每个 Agent 在独立 git 分支工作，合并时不会冲突

**实时协调**：Agent 之间通过 inbox 共享信息，不需要你当传话筒

## 09 高级功能

### 计划审批流程

Worker 提交计划，Leader 审批后再执行：

```bash
# Worker 提交计划
clawteam plan submit webapp architect "采用 JWT + Refresh Token 方案" --summary "双 Token 认证"

# Leader 审批
clawteam plan approve webapp <计划 ID> architect --feedback "LGTM，注意 Token 过期时间"

# 或者驳回
clawteam plan reject webapp <计划 ID> architect --feedback "请用 OAuth2 替代 JWT"
```

### 生命周期管理

```bash
# Worker 请求关闭（任务完成）
clawteam lifecycle request-shutdown webapp backend1 --reason "所有任务完成"

# Leader 批准关闭
clawteam lifecycle approve-shutdown webapp <请求 ID> backend1

# 或者拒绝
clawteam lifecycle reject-shutdown webapp <请求 ID> backend1 --reason "还有测试要修"
```

### 工作区管理

```bash
# 列出所有工作区
clawteam workspace list webapp

# 检查点（自动 commit 当前进度）
clawteam workspace checkpoint webapp backend1

# 合并到主分支
clawteam workspace merge webapp backend1

# 清理工作区
clawteam workspace cleanup webapp backend1
```

### 配置管理

```bash
# 查看配置
clawteam config show

# 修改配置
clawteam config set transport p2p
clawteam config set default_backend subprocess

# 健康检查
clawteam config health
```

### 环境变量覆盖

```bash
# 数据目录
export CLAWTEAM_DATA_DIR=/custom/path

# 传输方式
export CLAWTEAM_TRANSPORT=p2p

# 工作区模式
export CLAWTEAM_WORKSPACE=always

# 默认后端
export CLAWTEAM_DEFAULT_BACKEND=tmux

# 跳过权限确认
export CLAWTEAM_SKIP_PERMISSIONS=true
```

## 11 架构设计

```
人类："优化这个 LLM"
  │
  ▼
┌──────────────┐    clawteam spawn    ┌──────────────┐
│ 🦞 Leader    │ ────────────────────► │ 🤖 Worker    │
│ (Claude Code)│ ─────┐               │ (Claude Code)│
│              │      │               │ git worktree │
│ 使用：        │      │               │ tmux window  │
│ • spawn      │      │ clawteam spawn └──────────────┘
│ • task create│      │
│ • inbox send │      ▼               ┌──────────────┐
│ • board show │ ────────────────────► │ 🤖 Worker    │
│ • task wait  │      │               │ (Codex)      │
└──────────────┘      │               │ git worktree │
                      │               │ tmux window  │
                      │ clawteam spawn └──────────────┘
                      ▼
                ┌──────────────┐
                │ 🤖 Worker    │   每个 Worker 使用：
                │ (任意 CLI)   │   • task list（查看工作）
                │ git worktree │   • task update（汇报完成）
                │ tmux window  │   • inbox send（联系 Leader）
                └──────────────┘   • inbox receive（接收指令）
                      │
                      ▼
           ┌─────────────────────┐
           │    ~/.clawteam/     │
           │ ├── teams/   (谁)   │
           │ ├── tasks/   (什么) │
           │ ├── inboxes/ (沟通) │
           │ └── workspaces/    │
           │     (隔离代码)      │
           └─────────────────────┘
```

**所有状态都存储在 `~/.clawteam/` 的 JSON 文件中**。没有数据库，没有服务器，没有云端依赖。使用原子写入（tmp + rename）保证崩溃安全。

## 12 传输方式

| 传输方式 | 工作原理 | 适用场景 |
|---------|---------|---------|
| **file**（默认） | 收件箱目录中的 JSON 文件 | 单机、共享文件系统 |
| **p2p** | ZeroMQ PUSH/PULL + 文件降级 | 低延迟、跨机器 |

P2P 模式需要安装 ZeroMQ：
```bash
pip install -e ".[p2p]"
```

## 13 常见问题

### Q: Agent 卡在信任提示怎么办？

tmux 后端会自动确认 Claude Code 和 Codex 的信任提示。如果遇到其他 Agent 的权限问题，用 `--no-skip-permissions` 或设置环境变量：
```bash
export CLAWTEAM_SKIP_PERMISSIONS=false
```

### Q: 如何让 Agent 使用特定模型？

用 Profile 配置：
```bash
clawteam preset generate-profile moonshot-cn claude --name claude-kimi
clawteam spawn tmux --profile claude-kimi --team ...
```

### Q: 多个用户可以共享一个团队吗？

可以。ClawTeam 支持多用户工作流，通过命名空间隔离：
```bash
clawteam task list my-team --owner user1
```

### Q: 跨机器协作支持吗？

当前版本支持共享文件系统（NFS/SSHFS）或 P2P 传输。未来版本会加入 Redis 传输。

### Q: 如何调试问题？

```bash
# 查看团队状态
clawteam team status my-team

# 查看配置健康
clawteam config health

# 检查 Agent 是否正常运行
clawteam board show my-team
```

## 14 最佳实践

### 任务粒度

- 任务应该能在 1-2 小时内完成
- 太细：协调开销大
- 太粗：失去并行优势

### 依赖设计

- 尽量减少依赖链深度
- 理想情况：DAG 而非长链
- 关键路径任务优先分配

### 沟通频率

- Worker 完成阶段性工作后主动汇报
- 遇到阻塞立即发消息，不要硬扛
- Leader 定期查看看板，不要 micromanage

### 工作区清理

- 任务完成后及时合并和清理
- 避免工作区堆积占用磁盘
- 重要节点做 checkpoint
