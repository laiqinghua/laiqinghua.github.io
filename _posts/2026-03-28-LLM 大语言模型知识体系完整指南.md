# LLM 大语言模型知识体系完整指南

> 从 Token 基础到 Agent 实战，本文系统性梳理 LLM 大语言模型的核心概念、技术架构和实践方法，帮助开发者建立完整的知识体系。

## 一、LLM 基础概念

### 1.1 什么是大语言模型

**大语言模型（Large Language Model，简称 LLM）**是一种基于深度学习的人工智能系统，能够理解、生成和处理人类语言。

**核心能力：**
- 文本生成：自动创作文章、代码、对话等
- 语言理解：分析语义、情感、意图
- 知识推理：基于训练数据进行逻辑推理
- 多任务处理：翻译、摘要、问答等多种任务

**工作原理：** 基于 Transformer 架构，通过海量文本数据训练，学习语言的统计规律和语义关联。

### 1.2 LLM 发展历史

| 时间 | 事件 | 意义 |
|------|------|------|
| 2017 年 | Transformer 架构提出 | 奠定大模型技术基础 |
| 2018 年 | BERT 发布 | 开启预训练语言模型时代 |
| 2020 年 | GPT-3 发布 | 首个千亿参数规模的模型 |
| 2022 年 | ChatGPT 发布 | 大模型进入大众视野 |
| 2023 年 | GPT-4、Claude、Gemini 等模型涌现 | 大模型能力持续提升 |

**关键转折点：** 2017 年 Transformer 架构的提出，解决了传统 RNN/LSTM 的长距离依赖问题，为大模型发展奠定技术基础。

## 二、基础计量单位：Token

### 2.1 Token 的定义

**Token 是 LLM 处理文本的基本单位**，可以理解为"词元"或"语言片段"。

**通俗理解：**
- 英文：一个单词通常是 1 个 Token（如 "hello" = 1 Token）
- 中文：一个汉字通常是 1-2 个 Token（如 "你好" = 2-4 Token）
- 数字：连续数字可能被合并（如 "2026" = 1 Token）

### 2.2 Token 与字符的对应关系

| 语言类型 | 估算关系 | 示例 |
|----------|----------|------|
| 中文 | 1 汉字 ≈ 1.5 Token | "你好世界" ≈ 6 Token |
| 英文 | 1 单词 ≈ 1.3 Token | "Hello World" ≈ 3 Token |
| 数字 | 连续数字 ≈ 1 Token | "20260328" = 1 Token |
| 标点符号 | 通常 1 个 = 1 Token | "，" = 1 Token |

**经验法则：**
- 1000 Token ≈ 750 个英文单词
- 1000 Token ≈ 500-600 个汉字
- 1 万字中文文章 ≈ 15000-18000 Token

### 2.3 为什么需要理解 Token

**计费依据：** 大多数 LLM API 按 Token 收费

```
输入 1000 Token + 输出 500 Token = 1500 Token 费用
```

**性能优化：** 了解 Token 计算有助于优化 Prompt，降低成本

**上下文管理：** 理解 Token 限制，合理设计对话长度

## 三、核心概念：上下文（Context）

### 3.1 什么是上下文

**上下文（Context）** 是 LLM 在一次对话中能够"记住"的信息总量。

**类比理解：** 就像人的短期记忆，有容量限制。

### 3.2 上下文窗口（Context Window）

**定义：** 模型单次处理能接受的最大 Token 数量。

**包含内容：**
```
上下文窗口 = 系统提示词 + 用户输入 + 历史对话 + 模型输出
```

### 3.3 主流模型上下文窗口对比

| 模型 | 上下文窗口 | 约支持内容 |
|------|------------|------------|
| GPT-3.5 | 16K | 约 1.2 万字 |
| GPT-4 | 128K | 约 10 万字 |
| Claude 3 | 200K | 约 15 万字 |
| Gemini 1.5 Pro | 1M | 约 70 万字 |

**实际意义：** 上下文窗口越大，模型能"记住"的信息越多，适合处理长文档、长对话场景。

### 3.4 上下文超限怎么办

**问题：** 当输入内容超过上下文窗口时，模型会"遗忘"最早的信息。

**解决方案：**

**1. RAG 检索增强生成**
```
用户问题 → 检索相关知识 → 拼接知识 + 问题 → 发送给 LLM
```

**2. 摘要压缩**
```
长对话 → 提取关键信息 → 生成摘要 → 作为新对话的上下文
```

**3. 分块处理**
```
长文档 → 切分成多个块 → 分别处理 → 合并结果
```

## 四、CrossContext Window 框架

### 4.1 框架概述

**CrossContext Window** 是一种突破单模型上下文限制的技术框架。

**核心思想：** 将大问题拆解，通过多次调用 LLM 协同完成。

### 4.2 关键组件

**1. RAG（检索增强生成）**
```python
# 伪代码示例
def rag_query(user_question):
    # 1. 从知识库检索相关内容
    relevant_docs = vector_db.search(user_question, top_k=5)
    
    # 2. 拼接检索结果和问题
    context = "\n".join([doc.content for doc in relevant_docs])
    prompt = f"基于以下信息回答问题:\n{context}\n\n问题:{user_question}"
    
    # 3. 发送给 LLM
    return llm.generate(prompt)
```

**2. Prompt 优化**
- 结构化：使用清晰的格式组织信息
- 分层：将复杂任务拆解为多个步骤
- 角色设定：给 LLM 指定专业角色

## 五、Prompt Engineering 分层工程

### 5.1 Prompt 基础结构

**标准格式：**
```
[角色设定] + [任务描述] + [输入数据] + [输出要求] + [示例]
```

**示例：**
```
你是一位资深 Python 工程师。
任务：编写一个计算斐波那契数列的函数。
要求：
1. 使用递归实现
2. 添加类型注解
3. 包含单元测试

输入：n = 10
输出：函数代码和测试结果
```

### 5.2 分层方法论

**Layer 1：基础层 - 清晰表达需求**
```
❌ 模糊："帮我写个函数"
✅ 清晰："用 Python 写一个计算两个数最大公约数的函数"
```

**Layer 2：结构层 - 组织信息格式**
```markdown
## 任务
编写 HTTP 请求函数

## 输入
- URL: https://api.example.com
- 方法：GET
- 超时：30 秒

## 输出
- 响应状态码
- 响应数据（JSON 格式）
```

**Layer 3：约束层 - 添加限制条件**
```
约束条件：
- 必须使用 requests 库
- 需要异常处理
- 禁止使用全局变量
- 代码符合 PEP8 规范
```

**Layer 4：示例层 - 提供参考样例**
```
参考示例：
输入：get_data("https://api.com/users")
输出：{"status": 200, "data": [...]}
```

### 5.3 高级技巧

**1. Chain of Thought（思维链）**
```
请逐步思考：
1. 首先分析问题...
2. 然后确定解决方案...
3. 最后给出答案...
```

**2. Few-Shot Learning（少样本学习）**
```
示例 1:
输入："今天天气真好"
情感：正面

示例 2:
输入："我心情很糟糕"
情感：负面

请分析："这个产品太棒了"
情感：
```

**3. Role Prompting（角色设定）**
```
你是一位有 10 年经验的安全专家，
擅长代码审计和漏洞分析。
请审查以下代码的安全问题...
```

## 六、工具调用：MCP（Model Context Protocol）

### 6.1 什么是 MCP

**MCP（Model Context Protocol）** 是一种让 LLM 能够调用外部工具和 API 的标准协议。

**类比理解：** 就像给 LLM 装上了"手和脚"，让它能够执行实际操作。

### 6.2 为什么需要 MCP

**LLM 的局限性：**
- ❌ 无法访问实时数据（天气、股票、新闻）
- ❌ 无法执行代码
- ❌ 无法访问私有数据库
- ❌ 无法调用外部 API

**MCP 解决方案：**
- ✅ 定义工具接口
- ✅ 标准化调用方式
- ✅ 安全执行环境

### 6.3 MCP 工作原理

```
用户请求 → LLM 分析 → 识别需要工具 → 生成工具调用 → 执行工具 → 返回结果 → LLM 整合回答
```

### 6.4 MCP 工具定义示例

**JSON Schema 格式：**
```json
{
  "name": "get_weather",
  "description": "获取指定城市的当前天气",
  "parameters": {
    "type": "object",
    "properties": {
      "city": {
        "type": "string",
        "description": "城市名称，如'北京'、'上海'"
      }
    },
    "required": ["city"]
  }
}
```

**Python 实现：**
```python
from mcp import server

@server.tool("get_weather")
def get_weather(city: str) -> dict:
    """获取城市天气"""
    # 调用天气 API
    response = requests.get(f"https://api.weather.com/{city}")
    return {
        "city": city,
        "temperature": response.json()["temp"],
        "condition": response.json()["condition"]
    }
```

### 6.5 实际应用场景

**场景 1：查询实时信息**
```
用户："北京今天天气怎么样？"
→ LLM 调用 get_weather("北京")
→ 返回：{"temperature": 25, "condition": "晴"}
→ LLM 回答："北京今天晴天，气温 25 度"
```

**场景 2：执行代码**
```
用户："计算 1 到 100 的和"
→ LLM 调用 run_code("sum(range(1, 101))")
→ 返回：5050
→ LLM 回答："1 到 100 的和是 5050"
```

**场景 3：数据库查询**
```
用户："查询用户 ID 为 123 的订单"
→ LLM 调用 query_db("SELECT * FROM orders WHERE user_id = 123")
→ 返回订单数据
→ LLM 整合回答
```

## 七、相关概念：Agent（智能体）

### 7.1 什么是 Agent

**Agent（智能体）** 是能够自主规划、调用工具、完成复杂任务的 AI 系统。

**核心公式：**
```
Agent = LLM + 记忆 + 工具 + 规划能力
```

### 7.2 Agent 与 LLM 的区别

| 特性 | LLM | Agent |
|------|-----|-------|
| 能力 | 文本生成/理解 | 任务执行 |
| 记忆 | 单次对话上下文 | 长期记忆 |
| 工具 | 无法调用 | 可调用多种工具 |
| 规划 | 无 | 可拆解复杂任务 |
| 自主性 | 被动响应 | 主动执行 |

### 7.3 Agent 核心组件

**1. 感知（Perception）**
- 接收用户输入
- 理解任务目标
- 识别环境状态

**2. 规划（Planning）**
```
复杂任务 → 拆解为子任务 → 确定执行顺序 → 分配资源
```

**3. 记忆（Memory）**
- 短期记忆：当前对话上下文
- 长期记忆：向量数据库存储历史经验

**4. 工具（Tools）**
- API 调用
- 代码执行
- 文件操作
- 网络搜索

**5. 行动（Action）**
- 执行工具调用
- 生成响应
- 更新状态

### 7.4 Agent 技术架构

```
┌─────────────────────────────────────┐
│           用户接口层                 │
│   (Web/App/API/命令行)              │
└──────────────┬──────────────────────┘
               │
┌──────────────▼──────────────────────┐
│           Agent 核心层               │
│  ┌─────────────────────────────┐    │
│  │  规划器 (Planner)           │    │
│  │  - 任务拆解                 │    │
│  │  - 执行顺序                 │    │
│  └─────────────────────────────┘    │
│  ┌─────────────────────────────┐    │
│  │  记忆管理器 (Memory)        │    │
│  │  - 短期上下文               │    │
│  │  - 长期向量存储             │    │
│  └─────────────────────────────┘    │
│  ┌─────────────────────────────┐    │
│  │  工具注册表 (Tool Registry) │    │
│  │  - MCP 工具                 │    │
│  │  - API 封装                 │    │
│  └─────────────────────────────┘    │
└──────────────┬──────────────────────┘
               │
┌──────────────▼──────────────────────┐
│           执行层                     │
│   (LLM API + 工具执行环境)          │
└─────────────────────────────────────┘
```

### 7.5 Agent 应用场景

**场景 1：客服助手**
```
能力：
- 理解用户问题
- 查询订单系统
- 处理退换货请求
- 升级复杂问题到人工
```

**场景 2：数据分析助手**
```
能力：
- 连接数据库
- 执行 SQL 查询
- 生成可视化图表
- 输出分析报告
```

**场景 3：开发助手**
```
能力：
- 理解需求
- 编写代码
- 运行测试
- 修复 Bug
- 提交代码
```

## 八、任务类型：Agent Skill

### 8.1 什么是 Agent Skill

**Agent Skill** 是 Agent 能够执行的特定任务类型。

**类比理解：** 就像人的"技能"，如"会开车"、"会做饭"。

### 8.2 常见 Skill 类型

| Skill 类型 | 描述 | 示例 |
|-----------|------|------|
| 信息检索 | 搜索和整理信息 | 搜索新闻、查询资料 |
| 数据分析 | 处理和分析数据 | 统计、图表生成 |
| 内容创作 | 生成文本内容 | 写文章、写代码 |
| 任务自动化 | 执行重复任务 | 批量处理、定时任务 |
| 对话交互 | 多轮对话管理 | 客服、助手 |

### 8.3 Skill 定义示例

```yaml
skill:
  name: "weather_query"
  description: "查询指定城市的天气信息"
  trigger: ["天气", "气温", "下雨"]
  tools: ["get_weather_api"]
  steps:
    - 提取城市名称
    - 调用天气 API
    - 格式化返回结果
  output_format: "城市：{city}, 温度：{temp}°C, 天气：{condition}"
```

## 九、实践案例

### 案例 1：构建天气查询 Agent

**需求：** 创建一个能回答天气问题的助手

**步骤 1：定义工具**
```python
@mcp.tool
def get_weather(city: str, date: str = "today") -> dict:
    """获取天气信息"""
    api_key = os.getenv("WEATHER_API_KEY")
    response = requests.get(
        f"https://api.weather.com/v1/{city}",
        params={"date": date, "key": api_key}
    )
    return response.json()
```

**步骤 2：定义 Agent**
```python
class WeatherAgent:
    def __init__(self):
        self.llm = LLM(model="gpt-4")
        self.tools = [get_weather]
    
    def handle(self, user_input: str) -> str:
        # 分析意图
        intent = self.llm.analyze(user_input)
        
        if intent["type"] == "weather_query":
            city = intent["city"]
            weather = get_weather(city)
            return f"{city}今天{weather['condition']}，气温{weather['temp']}度"
        
        return self.llm.generate(user_input)
```

**步骤 3：测试**
```
用户："北京今天天气怎么样？"
Agent: "北京今天晴天，气温 25 度"

用户："上海会下雨吗？"
Agent: "上海今天多云转小雨，建议带伞"
```

## 十、总结

### 核心知识点回顾

| 概念 | 关键点 | 应用场景 |
