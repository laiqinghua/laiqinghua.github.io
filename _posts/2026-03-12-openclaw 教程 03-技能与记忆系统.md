上一篇讲了怎么把 OpenClaw 接入 WhatsApp、Telegram 这些消息平台。连上之后,AI 能接收你的消息、给你回复——但它还不够聪明。

你每次说"帮我回昨天那封邮件",它都得重新问你"什么邮件"；你说"按我惯用的风格写代码",它根本不知道你的风格是什么。

这一篇解决这两个问题：技能系统让 AI 知道怎么干复杂的活,记忆系统让 AI 真正记住你是谁。

## 01 工具 vs 技能：别搞混了

刚接触 OpenClaw 时,很多人会被"工具"和"技能"搞混。这里直接说清楚。

**工具（Tools）**

原子操作,一次只做一件事：读一个文件、发一条消息、搜索一个网页。用 TypeScript 或 Python 写的代码函数,AI 直接调用。

**技能（Skills）**

编排方案,告诉 AI 如何把多个工具串联起来完成一个完整任务。核心是一个 Markdown 文件,AI 读完就知道该怎么做了。

- 工具是厨房里的刀、锅、铲子——每个只做一件事
- 技能是菜谱——告诉你先切菜、再热油、然后翻炒、最后调味

举个具体例子。你说"帮我把今天的会议纪要发到 Slack 的 #team 频道",AI 背后要做这些事：

1. 搜索今天的日历事件（工具：`google_calendar_search`）
2. 找到会议纪要文件（工具：`file_search`）
3. 读取文件内容（工具：`file_read`）
4. 格式化为 Slack 消息（AI 自己处理）
5. 发送到指定频道（工具：`slack_send_message`）

每一步都是工具,但 AI 怎么知道要按这个顺序来？是技能告诉它的。

### 技能文件长什么样

每个技能的核心是一个 `SKILL.md` 文件（注意大写）。结构很简单：

```markdown
# 技能名称
描述和用途。

## 触发条件
什么情况下激活这个技能。

## 执行步骤
1. 第一步
2. 第二步

## 依赖工具
列出需要调用的工具。

## 注意事项
执行时的边界条件。
```

技能不需要编程,写 Markdown 就行。AI 读完这个文件,就知道该怎么处理对应的任务了——这是 OpenClaw 最聪明的设计之一。

### 技能的加载机制

50+ 个技能不会同时塞进 AI 的上下文（那会爆掉）。OpenClaw 用了按需加载策略：

```
用户消息进来
↓
Gateway 分析消息内容
↓
匹配相关技能（通过 triggers 关键词或语义匹配）
↓
将匹配的技能指令注入系统提示词
↓
AI 根据技能指令 + 工具列表执行任务
```

你说"帮我查邮件",只有 `gog` 技能会被加载；你说"帮我看 PR",只有 `github` 技能会被激活。其他技能安静地待在一边,不消耗 token。

## 02 内置技能概览

OpenClaw 内置了 50+ 个技能,覆盖办公、开发、生活各个场景：

| 分类 | 主要技能 |
|------|----------|
| 办公效率 | `gog`（Gmail+Calendar+Drive）、`notion`、`slack`、`trello` |
| 开发工具 | `coding-agent`、`github`、`gh-issues`、`tmux` |
| 笔记知识 | `obsidian`、`apple-notes`、`nano-pdf` |
| 多媒体 | `voice-call`、`peekaboo`（截图分析）、`camsnap`（摄像头识别） |
| 生活工具 | `weather`、`goplaces`、`things-mac` |

重点推荐：`coding-agent`

不是简单调几个工具,而是启动完整编程工作流：分析需求 → 读取相关代码 → 制定方案 → 编写代码 → 验证结果。能读写项目文件、执行 Shell 命令、同时修改多个文件。对开发者来说,这个技能基本可以替代大部分重复性编程工作。

## 03 技能的三种类型与 ClawHub 市场

| 类型 | 说明 | 位置 |
|------|------|------|
| bundled（内置） | 随 OpenClaw 安装,不可修改 | 安装目录 |
| managed（托管） | 从 ClawHub 安装的社区技能,可更新 | `~/.openclaw/workspace/skills/` |
| workspace（工作区） | 用户自己创建的技能 | `~/.openclaw/workspace/skills/` |

优先级：`workspace > managed > bundled`。你自己创建的同名技能会覆盖内置版本。

**ClawHub（clawhub.com）** 是 OpenClaw 的官方技能市场,类似手机应用商店,搜索、安装、发布社区技能都在这里。常用命令：

```bash
openclaw skills search "jira"           # 搜索技能
openclaw skills install skill-name      # 从 ClawHub 安装
openclaw skills install github:user/skill # 从 GitHub 安装
openclaw skills list                    # 查看已安装技能
openclaw skills update --all            # 更新所有技能
openclaw skills uninstall skill-name    # 卸载
```

## 04 技能组合：让 AI 自动帮你干活

单个技能已经很强大,但真正省心的用法是把多个技能串起来,变成自动化工作流。

### 自然语言描述工作流

最简单的方式是直接告诉 AI 你想要什么顺序：

```
每天早上 9 点：
1. 用 gog 查看今天的日历安排
2. 检查未读邮件,摘要一下
3. 用 github 看我负责的 PR 和 Issue 状态
4. 整理成"今日简报"发到 Slack #daily-standup 频道
```

AI 会自动依次调用 `gog`、`github`、`slack` 三个技能,串成一个完整的流程。

### 定时任务

通过 `openclaw.json` 配置定时任务,用标准 cron 表达式设置执行时间：

```json
{
  "schedules": [
    {
      "name": "morning-briefing",
      "cron": "0 9 * * 1-5",
      "agentId": "main",
      "prompt": "执行每日早报：查看日历、邮件、GitHub,整理成简报发到 Slack",
      "skills": ["gog", "github", "slack", "summarize"]
    },
    {
      "name": "weekly-review",
      "cron": "0 17 * * 5",
      "agentId": "main",
      "prompt": "生成本周总结：统计 PR、Issue、重要邮件,写入 Obsidian 周报",
      "skills": ["github", "gog", "obsidian"]
    }
  ]
}
```

上面配置了两个定时任务：工作日早 9 点自动生成简报,每周五下午 5 点自动写周报。设好之后完全不用管,AI 自己跑。

### 技能链式调用

自定义技能可以在 `SKILL.md` 里声明依赖其他技能,让技能调用技能：

```markdown
# 项目周报生成器

## 触发条件
当用户提到：项目周报、生成周报

## 依赖技能
- github
- gog
- obsidian

## 执行步骤
1. 用 github 技能收集本周合并的 PR、关闭的 Issue
2. 用 gog 技能收集本周会议记录和重要邮件
3. 整合数据,生成 Markdown 格式周报
4. 用 obsidian 技能保存到"周报"文件夹
```

你说一句"生成周报",AI 会自动调用三个技能依次完成,整个过程无需干预。

## 05 自己写一个技能

这是 OpenClaw 最有价值的功能之一。你可以教 AI 新的能力,而且大多数情况下只需要写 Markdown。

最省力的方式是直接告诉 AI：

```
帮我创建一个技能,用来管理我的读书笔记。
每次我说"记录读书笔记",就创建一个新的 Markdown 文件,
包含书名、作者、日期、关键摘录和我的感想。
```

AI 会自动用内置的 `skill-creator` 技能帮你生成完整的技能文件。如果你想完全控制,手动创建也不复杂。以"读书笔记管理"为例：

### 第一步：建目录

```bash
mkdir -p ~/.openclaw/workspace/skills/reading-notes
```

### 第二步：写 SKILL.md

```markdown
# 读书笔记管理

你是一个读书笔记助手。当用户要求记录读书笔记时,按以下流程操作。

## 触发条件
当用户提到以下关键词时激活：记录读书笔记、读书笔记、我读了一本书

## 依赖工具
- file_read
- file_write
- file_search

## 创建新笔记
1. 询问用户书名、作者、评分（如果未提供）
2. 在 workspace/reading-notes/ 目录下创建文件
3. 文件名格式：YYYY-MM-DD-书名.md
4. 使用模板填写内容
5. 更新 INDEX.md 索引
```

### 第三步：验证

```bash
openclaw skills list              # 确认技能被识别
openclaw skills check reading-notes # 检查就绪状态
```

然后在对话里测试：`记录读书笔记：《代码整洁之道》,作者 Robert C. Martin,5 星`

### 技能权限管理

每个技能能做什么,你说了算。在 `openclaw.json` 中配置：

```json
{
  "security": {
    "skillPermissions": {
      "autoApprove": ["file_read", "memory_read"],
      "requireConfirmation": ["file_write", "network"],
      "alwaysDeny": ["file_delete"],
      "shellExec": {
        "allowedCommands": ["git", "npm", "node"],
        "blockedCommands": ["rm -rf", "sudo"]
      }
    }
  }
}
```

低风险操作（读文件、读记忆）自动批准；中高风险操作（写文件、网络请求）每次都要你确认；危险操作（删文件）永远拒绝。

## 06 AI 为什么需要记忆

大语言模型本质上是无状态的。每次对话结束,它把你忘得一干二净。

没有记忆：

```
你：帮我改一下昨天那个 React 组件
AI：请问你指的是哪个组件？能提供更多上下文吗？
你：……我昨天跟你聊了两个小时啊
AI：抱歉,我没有之前对话的记录。
```

有记忆：

```
你：帮我改一下昨天那个 React 组件
AI：你说的是 UserProfile 组件吧？昨天你提到想把
   头像上传从 base64 改成 presigned URL,我来改一下。
```

记忆系统的价值在三个方面：

- **连续性** — 跨会话保持项目上下文,记住你正在做的事、进度和决策
- **个性化** — 知道你的代码风格、工具偏好、沟通习惯
- **效率** — 不用每次都重新解释技术栈、工作流程、团队结构

## 07 三层记忆架构

OpenClaw 的记忆分三层,各司其职：

```
┌─────────────────────────────────────┐
│ 会话记忆 (Session Memory)           │
│ 当前对话上下文,会话结束即消失       │
│ 容量受模型上下文窗口限制             │
└──────────────────┬──────────────────┘
                   │ 触发 compaction 时自动刷新
                   ▼
┌─────────────────────────────────────┐
│ 短期记忆 (Daily Logs)               │
│ memory/YYYY-MM-DD.md                │
│ 每天一个文件,自动加载今天 + 昨天     │
└──────────────────┬──────────────────┘
                   │ 手动或自动整理
                   ▼
┌─────────────────────────────────────┐
│ 长期记忆 (MEMORY.md)                │
│ 精选的持久信息                       │
│ 每次会话启动时自动加载               │
└─────────────────────────────────────┘
```

三层的分工很清晰：当前会话的内容放会话记忆,当天的临时笔记放日志,长期有效的信息放 MEMORY.md。

记忆文件的存放位置：

```
~/.openclaw/workspace/
├── MEMORY.md          # 长期记忆
├── USER.md            # 用户画像（偏好、习惯）
├── SOUL.md            # Agent 人格定义
├── memory/
│   ├── 2026-02-28.md  # 今天的日志
│   ├── 2026-02-27.md  # 昨天的日志
│   └── ...
└── sessions/
    └── *.jsonl        # 会话日志
```

### MEMORY.md：长期记忆

`MEMORY.md` 存的是精选的、持久有效的信息。每次 AI 启动都会加载这个文件：

```markdown
# 长期记忆

## 用户偏好
- 编程语言：TypeScript,不用分号
- 框架：Next.js 14 + React 18
- 代码风格：函数式优先,避免 class

## 项目信息
- 当前项目：电商平台重构
- 部署：Vercel（前端）+ Railway（后端）

## 重要决策
- 2026-02-20：从 REST 迁移到 tRPC
- 2026-02-22：支付方案选 Stripe
```

### USER.md：用户画像

`USER.md` 专门描述"你是谁",跟 `MEMORY.md` 的杂项记忆区分开来：

```markdown
# 用户画像

## 基本信息
- 姓名：老金 | 时区：Asia/Shanghai

## 技术偏好
- TypeScript（不用分号,单引号,2 空格缩进）
- React + Next.js | 后端：Node.js + Fastify

## 沟通偏好
- 喜欢直接,代码示例比文字解释有用
- 先给方案,再解释原因
```

开启 autoLearn

开启后,AI 会在对话中自动捕捉你的偏好并更新 `USER.md`。你说"改成箭头函数,我不喜欢 function 关键字",它会自动把这个偏好记下来,以后都照着做。

## 08 记忆的工作原理

### 向量搜索

记忆系统用 sqlite-vec 做向量存储——基于 SQLite 的轻量向量数据库,零配置、单文件,同时支持语义搜索。AI 搜索记忆时不是简单关键词匹配,而是语义理解。你问"我之前用什么支付方案",即使记忆里写的是"选了 Stripe 作为支付服务商",也能匹配到。

搜索能力包括（开箱即用,不需要配置）：

- **混合搜索** — 关键词匹配 + 向量语义搜索综合排序
- **时间衰减** — 越新的记忆权重越高
- **去重** — 避免返回高度相似的内容
- **查询扩展** — 自动扩展搜索词,提高召回率

### Compaction：上下文压缩

对话越来越长,上下文窗口快满了怎么办？OpenClaw 会触发 compaction（压缩）。压缩前,系统会先让 AI 把重要信息写入记忆文件,然后把早期消息压缩成摘要,释放空间继续对话：

```
正常对话：
[系统提示 | 记忆 | 消息 1 | 消息 2 | ... | 消息 N]

接近上限：
[系统提示 | 记忆 | [摘要：消息 1-50] | 消息 51 | ... | 消息 N]
```

你也可以手动触发：在对话中输入 `/compact` 命令。

## 09 怎么管理记忆

### 对话中直接操作

最自然的方式是直接跟 AI 说,它会根据内容自动决定写入哪个文件：

```
记住：我的 GitHub 用户名是 laojin-dev        # 写入 USER.md
记住：项目部署在 Hetzner 的 VPS 上           # 写入 MEMORY.md
记住：跟我说话不用太客气,直接说重点         # 写入 USER.md
我之前说过用什么支付方案来着？               # 查询
API 地址改了,新地址是 xxx                  # 更新
把关于旧项目 X 的记忆都删掉                 # 删除
帮我整理一下长期记忆,删掉过时的信息         # 整理
```

### CLI 命令

```bash
openclaw memory status    # 查看记忆索引状态
openclaw memory index     # 重建向量索引
openclaw memory search "部署"  # 语义搜索记忆
```

记忆文件是纯 Markdown,也可以直接用编辑器打开修改。改完后运行 `openclaw memory index` 重建索引就行。

## 10 记忆配置与安全

基础配置：

```json
{
  "agents": {
    "defaults": {
      "memory": {
        "enabled": true,
        "loadDays": 2,        // 启动时加载最近几天的日志
        "maxMemoryFileSize": 50000,
        "autoSave": true      // compaction 时自动保存重要信息
      }
    }
  }
}
```

不同 Agent 可以有不同配置：

```json
{
  "agents": {
    "main": { "memory": { "loadDays": 3 } },
    "coding": { "memory": { "loadDays": 1 } },
    "social": { "memory": { "enabled": false } }
  }
}
```

### 敏感信息保护

配置正则规则,AI 写入记忆时自动屏蔽敏感内容：

```json
{
  "memory": {
    "security": {
      "sensitivePatterns": ["\\b(sk-|api_key|password|secret)[\\w-]+\\b"],
      "action": "redact",
      "redactReplacement": "[REDACTED]"
    }
  }
}
```

无论有没有加密,密码、API Key、JWT Token 这类东西都不应该存进记忆。用环境变量或密钥管理器。高安全场景可以开启 AES-256-GCM 加密,记忆文件在磁盘上是密文,密钥通过环境变量提供。

## 11 知识库集成（RAG）

记忆文件适合存小量个人信息。如果你需要 AI 消化大量结构化文档（公司 Wiki、产品手册、代码库文档）,就需要 RAG（检索增强生成）。

- 传统方式：用户提问 → AI 凭记忆回答（可能瞎编）
- RAG 方式：用户提问 → 搜索知识库 → 找到相关文档 → AI 基于文档回答（有据可查）

OpenClaw 通过 `knowledge` 插件支持 RAG：

```json
{
  "knowledge": {
    "enabled": true,
    "sources": [
      {
        "name": "company-docs",
        "type": "directory",
        "path": "~/Documents/company-wiki",
        "glob": "**/*.md",
        "watch": true  // 文档更新后自动重新索引
      }
    ]
  }
}
```

知识源类型支持：本地目录、单文件、网页 URL、GitHub 仓库。

记忆 vs 知识库,一句话总结：记忆是 AI 在对话中产生的个人信息；知识库是你预先准备的参考文档。两者结合,才是完整的"有记忆的 AI"。

## 12 使用技巧

**技能方面**

- 写 SKILL.md 时步骤要编号,指令要具体——不要写"处理邮件",要写"搜索今天的未读邮件,按发件人分组,返回摘要"
- 给技能加使用示例（examples/ 目录）,能显著提高 AI 执行的准确率
- 权限最小化,只申请真正需要的权限
- 每个技能专注一件事,不要把太多功能堆在一起

**记忆方面**

- MEMORY.md 只放长期有效的信息,控制在 500 行以内
- 当天的临时笔记写日志文件,别往 MEMORY.md 塞
- 重要对话结束后,主动让 AI 记录要点："把今天讨论的技术方案要点记下来"
- 记忆文件是纯文本,可以用 Git 管理或定期打包备份

## 下一篇预告：多 Agent 协作与生产部署

现在你的 AI 已经会用技能、也记住你了。但目前只有一个 Agent 在跑——所有事都归它管,工作、生活、编程全混在一起,记忆也会越来越乱。

多 Agent 协作——让不同的 AI 助手各司其职

编程 Agent 只管代码,只加载 `coding-agent` 和 `github`；社交 Agent 只管消息平台；各自有独立的记忆,互不干扰。你的 WhatsApp 消息自动路由到社交 Agent,GitHub 通知自动路由到编程 Agent。

生产部署——从个人玩具升级成真正的服务

把 OpenClaw 跑在服务器上,7×24 小时不间断。进程管理、日志监控、故障恢复、Webhook 配置,让 AI 助手真正变成你的生产力基础设施。
